# Media Transcriber — 진화적 개발 로드맵 (Phase A/B)

목표 파이프라인: `미디어 파일 → WAV 추출 → VAD → (무음 skip) Qwen 전사 → SRT`

원칙: 각 단계는 **독립 실행 가능한 CLI 스크립트 + 눈으로 확인 가능한 산출물 + 명시적 통과 기준**을 갖는다.
다음 단계로 넘어간 뒤 문제가 생기면 이전 단계로 돌아올 수 있어야 하므로, 산출물은 재사용 가능한 형태(파일)로 남긴다.

---

## Stage 0. 테스트 셋 준비 (선행 작업, 코드 없음)

### 산출물
- 대표성 있는 짧은 클립 3~5개 (`testset/clip_01.mp4` ...)
  - 최소 구성: (a) 깨끗한 발화 1개, (b) 배경음/음악 섞인 것 1개, (c) 짧은 맞장구·감탄사("네", "어", "음")가 많은 것 1개, (d) 화자 2명 이상 겹치는 구간 1개
- 그중 1개 클립에 대한 **수동 정답 전사** (`testset/clip_01.reference.txt`) — 사람이 직접 듣고 작성

### 검증 방법론
- 검증 대상 아님 (기준 데이터 자체). 다만 "이 클립들이 실제 처리할 콘텐츠의 특성을 대표하는가"를 스스로 점검 — 너무 깨끗한 클립만 있으면 이후 단계 검증이 낙관적으로 왜곡됨.

## Stage 1. FFmpeg 추출

### 코드 산출물
- `extract_audio.py` (또는 파이프라인 공통 모듈의 함수)
  - 입력: 임의 미디어 파일 경로
  - 출력: `temp/{job_id}/audio_16k_mono.wav`
  - 로그: 원본 duration/포맷, ffmpeg 커맨드 전문, 종료 코드, 출력 WAV의 duration/샘플레이트/채널수

### 검증 방법론
- **자동 검증**: 출력 WAV를 다시 `soundfile.info()`로 열어 `sample_rate == 16000`, `channels == 1`, `duration ≈ 원본 duration`(±0.1초 이내)을 assert. 실패 시 즉시 에러로 드러나야 함 (침묵 실패 금지).
- **수동 검증**: 출력 WAV를 실제로 재생해서 원본과 내용이 같은지, 볼륨이 깨지지 않았는지 1회 귀로 확인.
- **통과 기준**: 테스트 셋 전체가 자동 검증을 통과하고, 1개 이상 수동 청취로 이상 없음 확인.

## Stage 2a. VAD Raw 확인

### 코드 산출물 
- `vad_raw_test.py` (1stage 프로그램을 확장)
- 입력: 임의 미디어 파일 경로
- 출력:
  - 콘솔 로그: raw segment 목록(시작/끝/길이), 구간 길이·침묵 길이 통계
  - `*_vad_raw.png`: 파형 + speech 확률 곡선 + raw segment 하이라이트

### 검증 방법론
- **정량**: 통계 출력 중 "1초 이하 침묵 개수", "1초 미만 구간 개수"가 얼마나 되는지 확인 — §12.2 병합이 왜 필요한지 감을 잡는 단계이지 여기서 통과/실패를 가르지는 않음.
- **정성 (핵심)**: PNG를 보면서
  - 사람이 들었을 때 말하고 있는 구간에 orange 하이라이트가 실제로 걸쳐 있는가
  - 명백한 무음 구간(배경음악만 있는 구간 포함)에 orange가 잘못 걸리지 않는가
  - 짧은 감탄사("네")가 통째로 누락되지 않는가
- **통과 기준**: "문장 단위 발화가 통째로 누락되는 경우 0건" (짧은 조각 단위 miss는 이 단계에서는 허용 — 병합 단계에서 흡수됨)
- threshold(기본 0.5)를 0.3/0.5/0.7 등으로 바꿔가며 같은 클립에 대해 비교 → 어떤 threshold가 통과 기준에 맞는지 기록해둔다 (§24 미결정 사항의 실측 근거로 재사용).

## Stage 2b. VAD 병합 후처리 (§12.2)

### 코드 산출물
- `vad_merge.py` (Stage 2a 결과를 입력으로 받아 병합 로직 적용)
  - 병합 규칙: 짧은 침묵(0.5~1초 이하) 병합 → 짧은 구간(1초 미만) 흡수 → 30초 초과 시 강제 분할 (§12.3 순서 그대로)
  - 파라미터를 CLI 인자로 노출 (`--min-silence 0.7 --min-speech 1.0 --max-chunk 30`) — 튜닝을 위해 하드코딩 금지
  - 출력: 병합된 segment 목록 + `*_vad_merged.png` (raw와 겹쳐서 비교 가능하게, 예: raw는 옅은 색, 병합 결과는 진한 색)
  - 로그: 병합 전/후 segment 개수, 30초 초과로 강제분할된 구간 개수

### 검증 방법론
- **정량**: 병합 후 segment 개수가 raw 대비 유의미하게 줄었는가(수백 개 → 수십 개 수준), 모든 segment가 30초 이하인가(assert).
- **정성**: raw vs merged PNG를 겹쳐 보면서, 병합 때문에 실제로는 분리돼야 할 서로 다른 화자/문장이 하나로 뭉개지지 않는지 확인.
- **통과 기준**: (a) 모든 병합 결과 구간이 30초 이하, (b) Stage 0 테스트 셋 전체에서 segment 개수가 실사용에 부담 없는 수준(체감 기준: 1분 오디오당 chunk 2~4개 내외가 자연스러움 — 너무 많으면 병합 파라미터가 너무 빡빡한 것)
- 여기서 결정된 파라미터 값을 `config.json`의 초기값으로 확정 (design.md §24 미결정 사항 해소).

## Stage 2c. Chunk 파일화 + manifest.json

### 코드 산출물
- `chunk_export.py`
  - Stage 2b의 최종 segment 목록을 받아 `temp/{job_id}/chunks/chunk_NNNN.wav`로 실제 분할 저장
  - `manifest.json` 생성 (design.md §14.2 스키마 그대로, status는 전부 `vad_extracted`)
  - 로그: 각 chunk 파일 크기/길이, 총 chunk 개수

### 검증 방법론
- **자동**: 모든 chunk wav의 duration 합이 원본 duration과 근사(무음 skip 정책에 따라 정확히 같지 않을 수 있음 — 이 프로젝트는 무음도 chunk 경계 판단에만 쓰고 chunk 자체는 발화 구간이므로, 합이 원본보다 작은 게 정상. 다만 "말하는 구간의 총합"과 근사한지 확인)
- **수동**: chunk 3~5개를 무작위로 골라 직접 재생 → 문장이 중간에서 잘리지 않았는지, 앞뒤로 다른 문장이 섞여 들어오지 않았는지 확인
- **통과 기준**: 무작위 샘플링한 chunk 전부에서 "문장이 중간에 잘림" 0건

> **여기까지가 사용자가 말한 "2번(VAD)"의 완성 지점.** Stage 0~2c 전부 통과해야 Phase B(전사)로 넘어간다.

---

## Stage 3. Qwen 전사 (무음 skip 적용)

### 코드 산출물
- `transcribe_chunks.py`
  - Stage 2c의 `manifest.json` + chunk wav들을 순회하며 llama-server(Qwen3-ASR)에 요청
  - **chunk마다 결과를 `chunk_NNNN.txt`로 저장** (design.md §13 구조 그대로 — 문제 있는 chunk만 재청취/재실험 가능하게)
  - 로그: chunk별 요청/응답 시간, 응답 원문 그대로(가공 전), 실패/타임아웃 여부

### 검증 방법론
- **정량 (Stage 0의 정답 전사가 있는 클립에 한해)**: CER 계산 (design.md §5A.4 정규화 방식 그대로 적용)
- **정성 (전체 테스트 셋)**: chunk_NNNN.txt를 chunk_NNNN.wav와 나란히 놓고 훑어보며
  - **할루시네이션**: 오디오에 없는 내용이 텍스트에 있는가 (특히 무음/노이즈 chunk에서 "감사합니다" 같은 상투어 출현 여부 — §5A.6에서 이미 보고된 패턴)
  - **무한반복**: 같은 단어/구절이 비정상적으로 반복되는가
  - **언어 오판**: §5A.8에서 확인된 것처럼 한국어를 일본어로 잘못 인식하는 사례가 있는가
- **통과 기준**: 테스트 셋 전체 chunk 중 할루시네이션/무한반복 발생 chunk 비율이 사전에 정한 허용치 이하 (예: 5% 이하 — 프로젝트 성격상 임의로 정하되, 정답 전사 있는 클립 CER과 함께 기록해서 다음 실험과 비교 가능하게)
- 문제 chunk가 발견되면: 해당 `chunk_NNNN.wav`만 따로 들어보고, VAD 경계 문제(Stage 2b/2c로 회귀)인지 프롬프트/파라미터 문제(Stage 3 내에서 조정)인지 원인을 구분해서 기록.

---

## Stage 4. SRT 생성

### 코드 산출물
- `build_srt.py`
  - `manifest.json`의 chunk start/end 시각 + `chunk_NNNN.txt` 전사 결과를 조합해 SRT 포맷으로 출력
  - 로그: 생성된 SRT 항목 수, 총 duration 대비 커버리지

### 검증 방법론
- **자동**: SRT 타임스탬프가 단조 증가하는지, 겹치는 구간이 없는지 assert
- **수동**: 실제 영상 플레이어(VLC 등)에 SRT를 얹어서 재생 — 자막이 뜨는 타이밍이 실제 발화 타이밍과 체감상 맞는지 확인 (이게 최종적으로 "타임스탬프 정합성"을 확인하는 가장 확실한 방법)
- **통과 기준**: 테스트 셋 전체에서 자막-음성 밀림이 체감상 없음 (기준: 육안상 0.5초 이상 밀리는 구간 0건)

## 전체 흐름 요약

| Stage | 코드 산출물 | 핵심 검증 수단 | 통과 기준(요약) |
|---|---|---|---|
| 0 | 테스트 클립 + 정답 전사 | 대표성 자체 점검 | - |
| 1 | `extract_audio.py` | duration/format assert + 청취 | 포맷 일치, 청취 이상 없음 |
| 2a | `vad_raw_test.py` (완료) | PNG 육안 확인 | 문장 단위 발화 누락 0건 |
| 2b | `vad_merge.py` | raw/merged PNG 비교 | 전체 30초 이하, chunk 수 적정 |
| 2c | `chunk_export.py` + manifest.json | 무작위 chunk 청취 | 문장 잘림 0건 |
| 3 | `transcribe_chunks.py` | CER + 육안 할루시네이션 점검 | 할루시네이션/반복 허용치 이하 |
| 4 | `build_srt.py` | 실제 플레이어 재생 확인 | 자막-음성 밀림 없음 |

각 Stage의 코드는 독립 스크립트로 유지하고, 이후 실제 GUI 앱(PySide6)에 통합할 때는 이 스크립트들의 함수를 그대로 모듈화해서 재사용하는 방향을 권장합니다 (재작성 대신 이식).
